## Issues Found & Fixed

### 1. No CSRF Protection on Any POST Form — FIXED ✅
**Severity:** High (security)
**Files:** `app.py`, all 10 form-bearing templates

Every state-changing form (login, register, create/delete dashboard, create/delete report,
add/delete data source) accepted POST requests with no Cross-Site Request Forgery protection.
An attacker hosting a malicious page could silently delete a victim's dashboards or data
sources simply by tricking them into loading an image or clicking a link.

**Fix — server side (`app.py`):**
- Added `import secrets` (constant-time comparison)
- Added `get_csrf_token()`: lazily creates a 64-hex-char token in the session
- Added `@app.before_request` hook `csrf_protect()`: rejects all POST/PUT/PATCH/DELETE
  requests whose `_csrf_token` form field does not `secrets.compare_digest`-match the
  session token. Returns 403 with a user-friendly error page.
- Registered `csrf_token` as a Jinja2 global so every template can call `{{ csrf_token() }}`
  without additional imports.
- Added `@app.errorhandler(403)` to serve a branded 403 page.

**Fix — templates (10 files):**
Added `<input type="hidden" name="_csrf_token" value="{{ csrf_token() }}" />` inside every
`<form method="POST">` element:
- `templates/login.html`
- `templates/register.html`
- `templates/dashboard_form.html`
- `templates/dashboards.html` (delete form per card)
- `templates/dashboard_view.html` (delete form)
- `templates/report_form.html`
- `templates/reports.html` (delete form per card)
- `templates/report_view.html` (delete form)
- `templates/data_source_form.html`
- `templates/data_sources.html` (delete form per row)

**Verified:** `curl -X POST http://localhost:5000/login` (no token) → **403**.
`GET /login` → **200**.

### 2. XSS via JavaScript Injection in `dashboard_view.html` — FIXED ✅
**Severity:** High (security)
**File:** `templates/dashboard_view.html:14`

The delete-confirmation dialog embedded the dashboard name directly into a JS string
inside an HTML attribute:

```html
onsubmit="return confirm('Delete dashboard "{{ dashboard.name }}"?')"
```

Jinja2 auto-escaping converts `'` → `&#x27;`, but the browser's HTML parser **decodes
entities before passing the value to the JS engine**. A dashboard named
`x'); fetch('/dashboards/1/delete',{method:'POST'});//` would therefore execute
arbitrary JavaScript. This was a stored XSS vector.

**Fix:** The name is now stored in a `data-name` HTML attribute (Jinja2 auto-escaping
makes this safe) and read at runtime via `this.dataset.name`. No user content ever
enters JS string context:

```html
<form ... data-name="{{ dashboard.name }}"
      onsubmit="return confirm('Delete dashboard &quot;' + this.dataset.name + '&quot;?')">
```

`this.dataset.name` is a DOM string read, not `eval`; no injection path exists.

### 4. Missing Server-Side Max-Length Validation on Username — FIXED ✅
**Severity:** Low (robustness)
**File:** `app.py` — `register` route

The registration route validated `len(username) >= 3` but had no upper-bound check.
The `User.username` column is `String(80)`, so a 100-character submission would cause
SQLAlchemy to raise an uncaught `DataError` at the DB layer rather than returning a
clean validation message.

**Fix:** Added `elif len(username) > 80: error = 'Username must be 80 characters or fewer.'`
after the min-length check, matching the HTML `maxlength="80"` attribute already present.

### 5. Missing `autocomplete` Attributes on Auth Forms — FIXED ✅
**Severity:** Low (UX / accessibility)
**Files:** `templates/login.html`, `templates/register.html`

Password managers and browsers use `autocomplete` hints to correctly fill and save
credentials. Without them, browsers fall back to heuristics and may offer to fill a
new-password field with a saved password (or vice versa).

**Fix:**
- `login.html`: `autocomplete="username"` and `autocomplete="current-password"`
- `register.html`: `autocomplete="username"`, `autocomplete="email"`, two fields with
  `autocomplete="new-password"`
