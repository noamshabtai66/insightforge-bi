## Issues Found & Fixed

### 1. CSP Blocked Inline Styles on Delete Forms — Critical (UI/Security)
**Files:** `templates/dashboards.html`, `templates/dashboard_view.html`,
`templates/data_sources.html`, `templates/reports.html`, `templates/report_view.html`

**Problem:** Five `<form>` elements used `style="display:inline"` to sit flush
inside flex containers alongside their sibling buttons. The app's own CSP header
(`style-src 'self'`) blocks inline `style=""` attributes in CSP Level 2+ browsers
(Chrome ≥ 25, Firefox ≥ 31, Safari ≥ 10). As a result, every delete button
rendered as a full-width block element, pushing the "Open" / "Back" buttons off
to a separate row, and the forms could not be dismissed visually without the
correct layout.

The `.form-inline { display: inline; }` CSS class was already defined in
`main.css` and served via `'self'`, but was never applied to these forms.

**Fix:** Replaced `style="display:inline"` with `class="form-inline"` on all
five affected form elements.

### 2. Missing SRI Integrity Hash on Chart.js CDN — Security
**File:** `templates/index.html`

**Problem:** The Chart.js 4.4.4 script was loaded from
`https://cdn.jsdelivr.net/npm/chart.js@4.4.4/dist/chart.umd.min.js` with no
`integrity` attribute. If the CDN were compromised or the unpinned tag silently
replaced, an attacker could serve arbitrary JavaScript that runs in users'
browsers.

**Fix:** Computed SHA-384 hash of the file and added `integrity` + `crossorigin`
attributes:
```html
integrity="sha384-NrKB+u6Ts6AtkIhwPixiKTzgSKNblyhlk0Sohlgar9UHUBzai/sgnNNWWd291xqt"
crossorigin="anonymous"
```

### 3. Deprecated SQLAlchemy 1.x Query API — Medium
**File:** `app.py`

**Problem:** Seventeen query calls used the legacy `Model.query.*` API
(`User.query.filter_by(...)`, `Sale.query.count()`, etc.). This API was
deprecated in SQLAlchemy 2.0 and emits `LegacyAPIWarning` in recent versions.
It will be removed in a future SQLAlchemy release.

**Fix:** Migrated all calls to the SQLAlchemy 2.x `db.session.scalar()`,
`db.session.scalars()`, `db.session.execute()`, and `db.first_or_404()` APIs.
Also replaced two remaining `db.session.query()` aggregate calls (in `index()`
and `api_kpis()`) with the modern `db.select(func.count()).select_from(Model)`
form.

### 4. Login Did Not Restore Originally Requested URL — UX
**File:** `app.py`, `templates/login.html`

**Problem:** When `login_required` redirected an unauthenticated user to
`/login`, the originally requested path was discarded. After a successful login,
users were always sent to `/` (Overview) regardless of which protected page they
were trying to reach.

**Fix:**
- `login_required` now appends `?next=<path>` to the login redirect.
- `login()` reads `next` from query string or POST body (forwarded via a hidden
  form field in `login.html`).
- `_safe_next()` validates the URL is a relative same-host path before using it,
  preventing open-redirect attacks.

---

### 5. 500 Error Handler Did Not Log the Exception — Observability
**File:** `app.py`

**Problem:** The `@app.errorhandler(500)` handler returned the error template
but never logged the underlying exception. In production this meant internal
errors produced no server-side trace, making debugging impossible without
additional middleware.

**Fix:** Added `logger.exception('Unhandled server error: %s', e)` before
rendering the template. `logger.exception` automatically appends the full
traceback to the log entry.

---

### 6. Email Field Accepted Any Arbitrary String — Validation
**File:** `app.py`

**Problem:** The registration route only checked that the email field was
non-empty; it performed no format validation. Users could register with values
like `"notanemail"` or `"@"` as their email address.

**Fix:** Added a module-level compiled regex `_EMAIL_RE = re.compile(r'^[^@\s]+@[^@\s]+\.[^@\s]+$')` and validated `email` against it before the
uniqueness check. The pattern accepts any `local@domain.tld` shape without
requiring an external dependency.
