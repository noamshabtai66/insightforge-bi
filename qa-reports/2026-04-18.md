## Issues Found & Fixed

### 1. XSS in Widget Delete Confirm Dialog (Medium — Security)

**File:** `templates/dashboard_view.html:75`

**Problem:** The widget delete `<form>` used inline Jinja2 interpolation to embed
the widget title directly into a JavaScript string inside the `onsubmit` event
handler:

```html
onsubmit="return confirm('Remove widget &quot;{{ w.title|replace('&quot;', '') }}&quot;?')"
```

Jinja2 auto-escaping converts `'` (single quote) to the HTML entity `&#39;`.
However, browsers decode HTML entities in attribute values *before* handing the
string to the JavaScript engine. This means `&#39;` becomes `'` in the executing
JS, breaking out of the string literal. A widget title such as
`'); alert(document.cookie);//` would execute arbitrary JavaScript when the user
clicked Delete.

The `|replace('&quot;', '')` filter only stripped the literal text `&quot;` from
the title before escaping — it did not prevent the entity-decode bypass.

Note: because widgets are user-scoped, this was a self-XSS for the current user.
Shared-dashboard or admin-view features planned for the roadmap would have
elevated this to a stored XSS against other users.

**Fix:** Use the same safe `data-name` / `this.dataset.name` pattern already
applied to the dashboard and report delete forms elsewhere in the codebase:

```html
<!-- Before (unsafe) -->
onsubmit="return confirm('Remove widget &quot;{{ w.title|replace('&quot;', '') }}&quot;?')"

<!-- After (safe) -->
data-name="{{ w.title }}"
onsubmit="return confirm('Remove widget \u201c' + this.dataset.name + '\u201d?')"
```

`this.dataset.name` is accessed as a DOM property — the browser decodes entities
to plain characters but the value is never interpreted as JS code.

### 2. Logout Button Rendered with Browser-Default Gray Background (Low — Visual)

**File:** `static/css/main.css`

**Problem:** `.btn-logout` styled a `<button>` element but omitted `background:
none` and `cursor: pointer`. All modern browsers apply a system-default
background color to `<button>` elements. With the dark theme, the logout button
appeared with an incongruous gray rectangle instead of a transparent background.
The `cursor: pointer` omission also made the button look non-interactive.

**Fix:**

```css
.btn-logout {
  /* added */
  background: none;
  cursor: pointer;
  /* existing properties unchanged */
}
```

### 3. Widget Grid Missing Bottom Margin; Widget Body Had No Minimum Height (Low — UX)

**File:** `static/css/main.css`

**Problem 1:** `.widget-grid` had no `margin-bottom`, so the widget list ran
flush against the page bottom or any element below it — inconsistent with all
other section containers which use 24 px spacing.

**Problem 2:** `.widget-body` had no `min-height`. When a Chart.js chart
initialises inside a zero-height container, the canvas collapses and the chart
renders invisible or as a 1 px line.

**Fix:**

```css
.widget-grid {
  margin-bottom: 24px;   /* added */
}

.widget-body {
  min-height: 180px;     /* added */
  position: relative;    /* added — required for Chart.js responsive sizing */
}
```

### 4. Six API Endpoints Using Deprecated `db.session.query()` Style (Low — Code Quality)

**File:** `app.py` (lines 646–765)

**Problem:** All six analytics API endpoints (`/api/sales-overview`,
`/api/revenue-by-region`, `/api/top-products`, `/api/sales-by-category`,
`/api/revenue-targets`, `/api/employee-stats`) used the SQLAlchemy 1.x legacy
ORM query interface (`db.session.query()`). SQLAlchemy 2.x marks this interface
as legacy and recommends `db.select()` with `db.session.execute()`. The rest of
the application already used the modern style exclusively, creating an
inconsistency and a deprecation-warning risk on future SQLAlchemy upgrades.

**Fix:** Migrated all six endpoints to the 2.x `db.select()` style. Example:

```python
# Before (legacy)
rows = (
    db.session.query(Region.name, func.sum(Sale.total_amount).label('revenue'))
    .join(Customer, Customer.region_id == Region.id)
    .join(Sale, Sale.customer_id == Customer.id)
    .group_by(Region.id)
    .order_by(func.sum(Sale.total_amount).desc())
    .all()
)

# After (SQLAlchemy 2.x)
rows = db.session.execute(
    db.select(Region.name, func.sum(Sale.total_amount).label('revenue'))
    .join(Customer, Customer.region_id == Region.id)
    .join(Sale, Sale.customer_id == Customer.id)
    .group_by(Region.id)
    .order_by(func.sum(Sale.total_amount).desc())
).all()
```

Row attribute access (`.name`, `.revenue`, etc.) is identical for both styles,
so no downstream changes were needed.

### 5. Connection String Field Missing `autocomplete="off"` (Info — Security Hygiene)

**File:** `templates/data_source_form.html`

**Problem:** The "Connection String" input (`type="text"`) could contain database
URIs with embedded passwords (e.g. `postgresql://user:secret@host/db`). Without
`autocomplete="off"`, browsers offer to save and autofill the value, potentially
exposing credentials via browser-stored form data or password managers that treat
it as a URL field.

**Fix:**

```html
<input type="text" id="connection_string" name="connection_string"
       class="form-control"
       placeholder="e.g. postgresql://user:pass@host/db"
       autocomplete="off" />
```
