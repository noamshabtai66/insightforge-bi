## Issues Found & Fixed

### 1. WSGI Startup Bug (Critical)
**File:** `app.py`  
**Problem:** `create_tables()` was only called inside `if __name__ == '__main__':`,
meaning database tables and the default admin account were never initialized when
the app was launched via a WSGI server (e.g. `gunicorn app:app`). The app would
crash on first request with `OperationalError: no such table`.  
**Fix:** Moved `create_tables()` call to module level so it runs on both
`python app.py` and WSGI import.

### 2. Admin Creation Race Condition (Medium)
**File:** `app.py`  
**Problem:** When multiple WSGI workers started simultaneously, two workers could
both pass the `if not User.query.first()` check and then both attempt to insert
the admin user, causing a `UNIQUE constraint failed` exception on the second
worker and leaving it in a broken state.  
**Fix:** Wrapped the `db.session.commit()` call in a `try/except` with rollback,
so the losing worker silently skips.

### 3. Missing HTTP 405 Error Handler (Medium)
**File:** `app.py`  
**Problem:** Flask would return an unhandled 405 response without rendering the
custom error template, inconsistent with the 403/404/500 handlers.  
**Fix:** Added `@app.errorhandler(405)` returning the `error.html` template.

### 4. Missing Blank Lines Between Route Handlers (Low — Code Style)
**File:** `app.py`  
**Problem:** No blank line between `forbidden()` and `not_found()` error handlers
(lines 96–99), and no blank line between `api_employee_stats()` and `api_kpis()`
(line 566–567). Violates PEP 8 two-blank-line rule between top-level functions.  
**Fix:** Added the missing blank lines.

### 5. `func.strftime` Used Without SQLite-Only Warning (Low)
**File:** `app.py`  
**Problem:** `api_sales_overview()` groups by month using `func.strftime('%Y-%m', ...)`,
which is SQLite-specific and silently fails on PostgreSQL.  
**Fix:** Added a docstring note with the PostgreSQL equivalent (`func.to_char`).

### 6. Missing Database Indexes on Hot Query Columns (Medium — Performance)
**File:** `models/models.py`  
**Problem:** All foreign key columns used in frequent filter/join queries lacked
database indexes. Affected columns:
- `Dashboard.user_id`, `Report.user_id`, `DataSource.user_id` — every list page
  filters by these.
- `Sale.customer_id`, `Sale.product_id` — used in all analytics joins.
- `Sale.sale_date` — used in date-range filters in both the dashboard and API.
- `Customer.region_id` — used in the regional revenue join.

On a 5 000-row `sale` table the impact is moderate; at production scale these
would cause full table scans.  
**Fix:** Added `index=True` to all seven columns.

### 7. Inline Styles in Templates (Low — Maintainability)
**Files:** `templates/dashboard_view.html`, `templates/report_view.html`  
**Problem:** Several `style="..."` attributes duplicated CSS already defined in
`main.css`, making layout changes require editing both files.  
**Fix:**
- `style="display:flex; gap:8px;"` → `class="flex-row-gap"` (both files)
- `style="max-width: 100%;"` → `class="full-width"`
- Inline grid `style="display:grid; …"` → `class="meta-grid"`
- Inline `style="margin-bottom:4px;"` → `class="meta-label"`
- Inline `style="padding:40px 0;"` → `class="empty-page-sm"`

### 8. Shared CHART_DEFAULTS Mutation Risk (Low — JS)
**File:** `static/js/charts.js`  
**Problem:** `CHART_DEFAULTS` was a single shared object. Chart.js can mutate
nested properties (`scales`, `plugins`) at runtime; spreading with `{ ...CHART_DEFAULTS }`
is a shallow copy and does not protect the nested objects. Multiple chart
instances could corrupt each other's configurations.  
**Fix:** Replaced the shared constant with two factory functions:
- `getChartDefaults()` — returns a fresh config object for cartesian charts.
- `getCircularTooltip()` — returns tooltip config for pie/doughnut/bar charts.
