## Issues Found & Fixed

### 1. Legacy SQLAlchemy 1.x API Still Present in seed.py — Medium
**File:** `seeds/seed.py` (lines 23, 91)

**Problem:** The 2026-04-14 report migrated all 17 legacy query calls in `app.py`
to the SQLAlchemy 2.x API, but two calls in the seed script were missed:
- `Region.query.first()` — used as the idempotency guard
- `Region.query.all()` — used when seeding revenue targets

These emit `LegacyAPIWarning` at runtime and will break on a future SQLAlchemy
version drop.

**Fix:**
```python
# Before
if Region.query.first():
for region in Region.query.all():

# After
if db.session.scalar(db.select(Region).limit(1)):
for region in db.session.scalars(db.select(Region)).all():
```

### 2. Missing HSTS Header — Security
**File:** `app.py` (`add_security_headers`)

**Problem:** The `add_security_headers` after-request hook set six security
headers but omitted `Strict-Transport-Security`. Browsers connecting over HTTPS
had no instruction to refuse future plain-HTTP connections, leaving sessions
vulnerable to SSL-stripping attacks in the initial request.

**Fix:** Added HSTS with a 1-year `max-age` and `includeSubDomains`. The header
is gated on `SESSION_COOKIE_SECURE` so it is only emitted when the application
is running in HTTPS mode (preventing broken redirects in plain-HTTP dev
environments):

```python
if app.config.get('SESSION_COOKIE_SECURE'):
    response.headers['Strict-Transport-Security'] = (
        'max-age=31536000; includeSubDomains'
    )
```

### 3. No Rate Limiting on Analytics API Endpoints — Security
**File:** `app.py` (all seven `/api/*` routes)

**Problem:** The seven JSON analytics endpoints (`/api/kpis`,
`/api/sales-overview`, `/api/revenue-by-region`, `/api/top-products`,
`/api/sales-by-category`, `/api/revenue-targets`, `/api/employee-stats`) each
required `@login_required` but carried no rate limit. An authenticated user
(or a compromised session) could hammer these endpoints continuously, exfiltrating
the full analytics dataset or exhausting the SQLite connection pool.

**Fix:** Added `@limiter.limit('120 per minute; 600 per hour')` to all seven
routes. The limit is generous for legitimate dashboard usage (a page load
triggers six parallel chart fetches) while blocking automated scraping.

### 4. Weak Password Policy — Security
**File:** `app.py` (register route), `templates/register.html`

**Problem:** The password minimum was 6 characters with no complexity requirement.
A 6-character all-lowercase password has ~309 million possible values — well
within brute-force range of even a modest offline attack.

**Fix:**
- Raised minimum length to 8 characters.
- Added a mandatory digit requirement (`any(c.isdigit() for c in password)`).
- Updated the `register.html` placeholder text and `minlength="8"` to match.

Verified three test cases:
- 7-char no-digit → rejected ("at least 8 characters")
- 8-char no-digit → rejected ("at least one number")
- 8-char with digit → accepted

### 5. Connection Strings Visible in Data Sources Table — Security / Privacy
**File:** `templates/data_sources.html`

**Problem:** The Data Sources list page displayed the first 20 characters of
each connection string (e.g. `postgresql://user:p…`). For a typical database
URL the username and the beginning of the password fall within the first 20
characters, leaking credentials to anyone with screen access or a shoulder
surfer.

**Fix:** Replaced the truncated display with a lock-icon placeholder:

```html
<!-- Before -->
{{ s.connection_string[:20] }}{% if s.connection_string|length > 20 %}…{% endif %}

<!-- After -->
<span title="Connection string configured — value hidden for security">&#128274; [configured]</span>
```

The tooltip clarifies the field is intentionally hidden.

### 6. No Pagination on List Views — Performance / UX (was remaining TODO)
**Files:** `app.py`, `templates/dashboards.html`, `templates/reports.html`,
`templates/data_sources.html`, `static/css/main.css`

**Problem:** The three list routes (`/dashboards`, `/reports`, `/data-sources`)
loaded all of the user's records in a single query with no upper bound. At scale
this would cause memory pressure on the server and slow page loads. This was
listed as a medium-priority remaining TODO since 2026-04-09.

**Fix:**
- Added a `PER_PAGE = 20` constant at module level in `app.py`.
- All three list routes now:
  1. Accept an optional `?page=N` query parameter (default 1, clamped to valid range).
  2. Count total records with `func.count()` to derive `total_pages`.
  3. Apply `.limit(PER_PAGE).offset((page - 1) * PER_PAGE)` to the record query.
  4. Pass `page` and `total_pages` to the template.
- Each template renders a `<nav class="pagination">` control only when
  `total_pages > 1`, keeping the UI clean for users with few records.
- Added a `.pagination` CSS block to `main.css` with styled prev/next arrows
  and highlighted current-page indicator.
