## Issues Found & Fixed

### 1. Double DB Query Per Request — FIXED
**Severity:** Medium (performance)
**File:** `app.py` — `login_required` decorator + `current_user()`

Every protected route was making two separate `db.session.get(User, ...)` calls: once
inside `login_required` (to check for stale sessions) and again inside each route via
`current_user()`. On a busy server with 100 routes/second this wastes 100 extra DB
round-trips per second.

**Fix:** The fetched user is now stored in `flask.g` by `login_required`:
```python
g.current_user = user
```
`current_user()` checks `g` first before hitting the DB:
```python
def current_user():
    if hasattr(g, 'current_user'):
        return g.current_user
    ...
```
All protected routes now pay exactly one DB query per request for user lookup.

### 2. `new_report` Redirected to List Instead of New Report View — FIXED
**Severity:** Low (UX inconsistency)
**File:** `app.py:344`

`new_dashboard` redirected to `view_dashboard` (the created resource), but `new_report`
redirected to the `/reports` list page. Inconsistent UX and forces an extra click to
open the newly created report.

**Fix:** Changed redirect target:
```python
# Before
return redirect(url_for('reports'))
# After
return redirect(url_for('view_report', report_id=report.id))
```

### 3. `SESSION_COOKIE_SECURE` Not Configured — FIXED
**Severity:** Medium (security)
**File:** `app.py`

The `Secure` cookie flag was absent entirely, meaning session cookies could be sent over
unencrypted HTTP in production. Added configurable via environment variable:
```python
app.config['SESSION_COOKIE_SECURE'] = os.environ.get('SESSION_COOKIE_SECURE', 'false').lower() == 'true'
```
Default is `false` (for local HTTP dev); set `SESSION_COOKIE_SECURE=true` in production
to enforce HTTPS-only cookie transmission.

### 4. Missing `Permissions-Policy` Header — FIXED
**Severity:** Low (security hardening)
**File:** `app.py` — `add_security_headers()`

The `Permissions-Policy` header was absent, leaving browser feature access (camera,
microphone, geolocation, payment) unrestricted. Added a restrictive policy:
```
Permissions-Policy: camera=(), microphone=(), geolocation=(), payment=()
```
This is a defence-in-depth measure for a BI platform that does not use these APIs.

### 5. `print()` Used Throughout `app.py` — FIXED
**Severity:** Low (code quality / ops)
**File:** `app.py`

`print()` bypasses Python's logging infrastructure, making it impossible to control
log level, format, or destination without code changes. All `print()` calls replaced
with the `logging` module:
- Added `import logging` and `logger = logging.getLogger(__name__)`
- Configured `logging.basicConfig()` at startup with timestamp + level format
- `SECRET_KEY` warning: `print(..., file=sys.stderr)` → `logger.warning(...)`
- Admin creation message: `print(...)` → `logger.info(...)`

### 6. Delete Button Missing from `dashboard_view.html` — FIXED
**Severity:** Low (UX gap)
**File:** `templates/dashboard_view.html`

The dashboard detail view had no delete action — users had to navigate back to the
dashboards list to delete a dashboard. Added a delete button alongside the Back button
in the page header, consistent with the pattern in `report_view.html`:
```html
<form method="POST" action="{{ url_for('delete_dashboard', ...) }}"
      onsubmit="return confirm('Delete dashboard ...')" style="display:inline">
  <button type="submit" class="btn btn-danger btn-sm">Delete</button>
</form>
```

### 7. Missing `maxlength` HTML Attributes on Form Inputs — FIXED
**Severity:** Low (UX / robustness)
**Files:** `templates/dashboard_form.html`, `templates/report_form.html`,
           `templates/data_source_form.html`, `templates/register.html`

Server-side validation enforced 120-character limits on name fields, but the HTML inputs
lacked `maxlength` attributes, so users could type unlimited characters before receiving
a server-side error. Added `maxlength` matching DB column constraints:
- `name` fields: `maxlength="120"`
- `username`: `maxlength="80"` (matches `User.username` column)
- `email`: `maxlength="120"` (matches `User.email` column)

### 8. Mobile Nav Completely Hidden with No Alternative — FIXED
**Severity:** Medium (accessibility / UX)
**Files:** `templates/base.html`, `static/css/main.css`, `static/js/main.js`

At ≤768px the nav links were `display:none` with no replacement UI, making the app
unusable on phones and tablets. Implemented a hamburger menu:
- Added a `<button class="nav-hamburger">` with three span bars to `base.html`
- CSS: button visible only on mobile, adds `.open` class to nav-links for toggle
- JS: `click` handler on hamburger toggles `.open` and updates `aria-expanded`
- Hidden `nav-username` on mobile to save space; logout button still visible
