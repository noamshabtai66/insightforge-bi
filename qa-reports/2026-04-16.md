## Issues Found & Fixed

### 1. Logout Route Accepts GET — CSRF Vulnerability (Medium)
**File:** `app.py:273`, `templates/base.html:26`

**Problem:** The `/logout` route responded to HTTP GET, making it trivially
exploitable via a CSRF logout attack. Any page containing
`<img src="/logout">` or a zero-pixel iframe pointing at `/logout` would silently
log out the victim. While this does not grant the attacker access to the account,
it is a denial-of-service on the user's session and a well-documented CWE-352
finding that would fail most security audits.

**Fix:**
- Changed the route decorator from `@app.route('/logout')` (implicit GET) to
  `@app.route('/logout', methods=['POST'])`.
- Replaced the `<a href="/logout">` link in `templates/base.html` with a
  `<form method="POST">` containing the existing CSRF token input. The
  `btn-logout` CSS class was already styled to work on `<button>` elements,
  so no visual change occurs.
- The existing `csrf_protect` before-request hook now validates the CSRF token
  on logout, providing protection against both logout CSRF and replay attacks.

```python
# Before
@app.route('/logout')
def logout(): ...

# After
@app.route('/logout', methods=['POST'])
def logout(): ...
```

```html
<!-- Before -->
<a href="{{ url_for('logout') }}" class="btn-logout">Logout</a>

<!-- After -->
<form method="POST" action="{{ url_for('logout') }}" class="form-inline">
  <input type="hidden" name="_csrf_token" value="{{ csrf_token() }}" />
  <button type="submit" class="btn-logout">Logout</button>
</form>
```

### 2. Widget Management UI Missing (Medium — roadmap item)
**Files:** `app.py`, `templates/dashboard_view.html`, `static/js/charts.js`,
`static/css/main.css`

**Problem:** The `Widget` model and DB table have existed since the initial
schema, but there was no way for users to add or delete widgets. The dashboard
view displayed a hard-coded "coming soon" message. This was the highest-priority
remaining TODO since 2026-04-14.

**Fix:** Implemented full widget add/delete functionality:

**app.py:**
- Added `ALLOWED_WIDGET_TYPES = {'bar', 'line', 'pie', 'doughnut'}` allowlist.
- Added `WIDGET_DATA_SOURCES` mapping form keys to API endpoint paths for the
  four endpoints that return the `{labels, revenue}` format.
- Added `POST /dashboards/<id>/widgets/new` — validates title (≤120 chars),
  widget type (allowlist), data source (allowlist), auto-assigns `position`,
  stores JSON config with `api_endpoint`.
- Added `POST /dashboards/<id>/widgets/<wid>/delete` — verifies dashboard
  ownership before deleting.
- Updated `view_dashboard` to parse each widget's JSON config and pass
  `widget_configs` dict to the template.

**templates/dashboard_view.html:**
- Replaced "coming soon" empty state with a `<details>`/`<summary>` collapsible
  form ("+ Add Widget"). Using `<details>` avoids inline JavaScript and is
  fully CSP-compliant.
- Widget cards now include a `×` delete button (POST form with CSRF token).
- Canvas elements carry `data-chart-api` and `data-chart-type` HTML attributes
  so the JavaScript can discover and initialise them without any inline script.

**static/js/charts.js:**
- Added `initWidgetChart(canvasId, chartType, apiEndpoint)` — fetches data from
  the specified endpoint and renders a Chart.js chart of the requested type
  (line/bar/pie/doughnut) with the same colour palette and tooltip styles as
  the overview page charts.
- Extended the existing `DOMContentLoaded` listener to auto-discover all
  `canvas[data-chart-api]` elements and call `initWidgetChart` for each.
  The overview page charts are unaffected (their canvases have no `data-chart-api`
  attribute).

**static/css/main.css:**
- Added `.add-widget-details`, `.add-widget-form`, and `.add-widget-fields`
  styles for the collapsible add-widget panel (responsive 4-column grid on
  desktop, 1-column on mobile).

### 3. `updated_at` Timestamps Missing on Mutable Models (Low — roadmap item)
**File:** `models/models.py`

**Problem:** `Dashboard`, `Report`, and `DataSource` models had `created_at`
but no `updated_at`. Any future edit feature would have no way to record when
a record was last changed.

**Fix:** Added `updated_at` column to all three models:

```python
updated_at = db.Column(db.DateTime, default=_utcnow, onupdate=_utcnow)
```

`default=_utcnow` populates the column on INSERT; `onupdate=_utcnow` fires
SQLAlchemy's column-level `ON UPDATE` hook on any subsequent flush that touches
the row.

**Migration note:** `db.create_all()` does not alter existing tables. Fresh
databases will have the column automatically. Existing development databases
require either a `DROP + recreate` or an Alembic migration:
```sql
ALTER TABLE dashboard    ADD COLUMN updated_at DATETIME;
ALTER TABLE report       ADD COLUMN updated_at DATETIME;
ALTER TABLE data_source  ADD COLUMN updated_at DATETIME;
```

### 4. No Password Strength Feedback on Registration Form (Info — roadmap item)
**Files:** `templates/register.html`, `static/js/main.js`, `static/css/main.css`

**Problem:** The registration form accepted any password that passed the 8-char /
digit server-side check but gave no real-time feedback to help users choose a
strong password. The placeholder text was the only hint.

**Fix:**
- Added a `<div class="password-strength">` block beneath the password input
  in `register.html` containing a fill-bar div and a strength label span.
- Added `scorePassword(pw)` to `main.js`: awards points for length ≥8, length
  ≥12, presence of a digit, and presence of a special character, maps the
  0–3 score to `weak / fair / good / strong`.
- The `DOMContentLoaded` handler in `main.js` now wires an `input` event on
  `#password` (guarded by element existence so it is a no-op on non-register
  pages) that updates the fill bar's width and colour class in real time.
- Added CSS in `main.css`: `.password-strength` (hidden until `.visible` class
  added by JS), `.strength-bar` / `.strength-fill` with four colour states
  (red / amber / blue / green), `.strength-label` with matching colours.
- No inline styles are used; the implementation is fully compliant with the
  existing `style-src 'self'` Content Security Policy.
