## Issues Found & Fixed

### Bug — Fixed

| # | Severity | Issue | Fix |
|---|----------|-------|-----|
| 1 | MEDIUM | **`“`/`”` rendered as literal text in widget delete dialog** — `dashboard_view.html` line 74 contained the six-character ASCII sequences `“` and `”` (Python string escapes). Jinja2 does not interpret `\u` escapes in template files, so the confirm dialog displayed `Remove widget “Title”?` verbatim instead of curly-quoted text. | Replaced `“`/`”` with `&quot;` HTML entity, consistent with every other delete-confirm form in the codebase (`dashboards.html`, `reports.html`, `data_sources.html`). Verified with a hex-dump of the template file before and after. |

### Security — Fixed

| # | Severity | Issue | Fix |
|---|----------|-------|-----|
| 2 | MEDIUM | **Connection string input rendered as plain text** — `data_source_form.html` used `type="text"` for the connection string field. Although `autocomplete="off"` was set, `type="text"` still allows the value to appear in browser history replay and screen-sharing sessions. | Changed input to `type="password"` so the value is masked on-screen and excluded from browser autocomplete history. `autocomplete="off"` retained as a belt-and-suspenders measure. |
| 3 | LOW | **No rate limit on `new_widget` POST route** — unlike all other mutating routes, `new_widget` had no `@limiter.limit` decorator, allowing an authenticated user to spam widget creation without any throttle. | Added `@limiter.limit('30 per minute; 200 per hour')` decorator, consistent with the limit applied to analytics API endpoints. |

### UX — Fixed

| # | Severity | Issue | Fix |
|---|----------|-------|-----|
| 4 | MEDIUM | **Dashboard and report forms blanked on validation error** — `new_dashboard()` and `new_report()` cleared all fields when the server rejected a submission (name too long, description too long, etc.), forcing users to retype. This was inconsistent with the register form fix applied on 2026-04-30. | Both routes now pass `name` and `description` back to their templates on every error path. `dashboard_form.html` and `report_form.html` each gain `value="{{ name or '' }}"` on the name input and `{{ description or '' }}` content on the textarea. |
| 5 | LOW | **Data source form lost `name` and `type` on validation error** — same root cause as #4 but for `new_data_source()`. Connection string is deliberately not repopulated (sensitive). | `new_data_source()` now passes `name` and `data_type` back on error. `data_source_form.html` gains `value="{{ name or '' }}"` on the name input and `selected` attributes on each `<option>` keyed off `data_type`. |

### Code Quality — Fixed

| # | Severity | Issue | Fix |
|---|----------|-------|-----|
| 6 | LOW | **`import re` placed mid-file between third-party imports** — `import re` appeared at line 28, after `from flask_limiter import …` and before `from flask import …`, breaking PEP 8's import grouping convention (stdlib → third-party → local). | Moved `import re` into the stdlib block at the top of `app.py` alongside `import csv`, `import io`, etc. Removed the dangling `import re` from line 28. |

### Portability — Fixed

| # | Severity | Issue | Fix |
|---|----------|-------|-----|
| 7 | MEDIUM | **`api_sales_overview` used SQLite-specific `func.strftime`** — noted as a TODO since 2026-04-30. Prevented a drop-in switch to PostgreSQL without code changes. | Added `_month_trunc(col)` helper in `app.py` that returns `func.strftime('%Y-%m', col)` for SQLite and `func.to_char(col, 'YYYY-MM')` for PostgreSQL, branching on `db.engine.dialect.name`. `api_sales_overview` now uses `month_col = _month_trunc(Sale.sale_date)` in the `select`, `group_by`, and `order_by` clauses. |

## What Was Already Good (unchanged)

- CSRF protection on all state-changing routes
- Rate limiting on login (20/min), register (10/min), all analytics APIs (120/min), export (10/min)
- Account lockout after 5 failed logins (15-minute lock)
- Security headers: CSP, X-Frame-Options, Referrer-Policy, HSTS (production), Permissions-Policy

- Open redirect prevention (`_safe_next`)

- Session fixation mitigation (`session.clear()` on login)

- Streaming CSV export (`yield_per(500)`)
- Smart pagination (`_page_range` with ellipsis)
- Fernet connection-string encryption (when `CONNECTION_KEY` is set)
- `ensure_columns()` startup schema migration

## Remaining TODOs

### High Priority

- **Audit log table** — add `AuditLog(user_id, action, resource_type, resource_id, ip, ts)`
  and write an entry on every create/update/delete operation. This is the single biggest
  observability gap in the application.
- **`CONNECTION_KEY` rotation script** — once data is encrypted with a Fernet key, re-keying
  requires iterating rows and re-encrypting with the new key. Add
  `scripts/rotate_connection_key.py`.

### Medium Priority

- **Host Chart.js locally** — eliminates CDN network dependency (SRI mitigates supply-chain
  risk but local hosting removes the outbound request and the CDN availability dependency).
- **Widget JSON schema validation on read** — `Widget.config` is server-generated from a
  whitelist but deserializing on read without schema validation leaves a gap for future
  code paths.

### Low Priority

- **Admin RBAC** — `User.is_admin` exists but no route checks it. Add an `admin_required`
  decorator and an `/admin` management view.
- **Unit / integration tests** — zero test coverage. Add `pytest` fixtures for route smoke
  tests and critical model methods.
- **Pagination count-query caching** — each paginated page runs a `COUNT(*)` query; for
  large tables a short-lived cache (e.g., 30 s TTL) would reduce DB load.
